# portool v0.1 実装計画

仕様書: `docs/spec-v0.1.md`(凍結済み・実装着手可能レベル)。本計画は仕様を 4 つの逐次タスクに分解する。各タスクは fresh subagent が実装し、コミットまで行う。

## Global Constraints(全タスク共通・仕様からの凍結値)

- クレート名/バイナリ名: `portool`。edition 2021。ライセンス MIT。リポジトリ https://github.com/t09tanaka/portool
- 依存クレート: `clap`(v4, derive) / `serde`(derive) / `serde_json` / `toml`(v0.8) / `fs2` / `sha2` / `chrono`。ランタイムの atomic write に `tempfile` 可。dev-deps に `tempfile`
- デフォルト設定: pool `[3000, 9999]`、`subrange_size = 500`、`block_align = 5`、`gc_days = 30`
- ブロックサイズ = `max(最大オフセット+1, 宣言数)` を `block_align` の倍数に切り上げ。マニフェスト不在時はサイズ = `block_align`、env 出力は `PORT=<block_start>` の 1 行のみ
- `manifest_hash` = ファイルバイト列の SHA-256 hex 先頭 12 文字。不存在は `null`
- env 変数名 = キーを大文字化 + `_PORT`(`web` → `WEB_PORT`)。キーの正規表現 `[a-z][a-z0-9_]*`、オフセット重複はエラー(exit 1)
- `.env.portool` 形式(バイト単位で厳密に、末尾改行あり、行はオフセット昇順):
  ```
  # generated by portool — DO NOT EDIT
  # block: 3000-3004  project: esimdb  worktree: /home/takuto/dev/esimdb
  WEB_PORT=3000
  ```
  (`—` は U+2014。`block:` 等の区切りはスペース 2 個)
- スロット選択: branch が `main`/`master` → slot 0、それ以外 → `FNV-1a-32(branch ?? worktree_path) % slots`、探索は preferred から mod で一周。FNV-1a-32: offset_basis 2166136261, prime 16777619
- exit codes: 0 成功(no-op 含む) / 1 一般エラー / 2 サブレンジ枯渇から回復不能 / 3 プール枯渇 / 4 flock タイムアウト(10s)
- 台帳パス `${XDG_STATE_HOME:-~/.local/state}/portool/registry.json`、ロック `registry.json.lock`、設定 `${XDG_CONFIG_HOME:-~/.config}/portool/config.toml`
- 台帳スキーマは仕様 §5 の JSON 例のとおり(version=1, range, projects{name, subranges, worktrees{block, branch, manifest_hash, pinned, label, allocated_at, last_seen_at}}, reservations[{block, label, pinned}])
- タイムスタンプはローカルタイムゾーン付き RFC 3339 秒精度(例 `2026-07-15T10:00:00+09:00`)
- 台帳・env の書き込みは必ず temp ファイル + rename(同一ディレクトリ内)
- 外部プロセス呼び出しは `git rev-parse` 系と `git worktree list --porcelain` のみ。git2 クレート禁止
- fast path(sync)はロックを取らない read-only。エントリ存在 + manifest_hash 一致 + `.env.portool` バイト一致 → 何も書かず exit 0

### 凍結した実装判断(仕様の曖昧点の解決)

1. **ブランチ検出**: `git symbolic-ref --short -q HEAD`。失敗(detached / unborn 以外)や空出力 → `branch: null`。unborn branch(コミット 0 の新規リポジトリ)では symbolic-ref が名前を返すのでそれを使う
2. **フック設置場所**: `<common_dir>/hooks/post-checkout`(全 worktree 共有)。`core.hooksPath` は v0.1 では考慮しない
3. **複数サブレンジでのブロック探索**: subranges を配列順に走査。各サブレンジ内で `slots = floor(width / block_size)`(0 なら skip)、preferred スロットから mod 一周。全サブレンジで不発 → 新サブレンジ確保して再試行
4. **exit 2 と 3 の境界**: サブレンジ確保(初回・追加とも)でプールに空きなし → exit 3。新サブレンジを確保してもブロックが置けない(`block_size > subrange_size` 等)→ exit 2(`block_size > subrange_size` は確保前に検知して即 exit 2)
5. **ブロック重複チェック**: 全プロジェクトの全ブロック + reservations に対してグローバルに行う(安全側)
6. **`init` のフラグ**: `--hook-only` → フック設置のみ、`--gitignore-only` → .gitignore 追記のみ、無指定 → 両方 + `sync` 実行。2 フラグは排他(clap の conflicts_with)
7. **`.gitignore` 追記先**: カレント worktree ルートの `.gitignore`。`.env.portool` と完全一致する行が既にあれば no-op
8. **フック追記の冪等判定**: 既存 post-checkout に `portool sync` を含む行があれば no-op。無ければ `command -v portool >/dev/null 2>&1 && portool sync --quiet` を末尾に追記。新規作成時は仕様 §10.1 の 3 行スクリプト。実行ビット(0o755)を保証
9. **`sync` の出力**: fast path は無出力。slow path は変更内容 1 行を stdout(例 `portool: allocated 3000-3004 for /path/to/worktree`)、`--quiet` で抑制。警告(hint 等)は常に stderr
10. **フック未設置警告**: `<common_dir>/hooks/post-checkout` が無い、または `portool sync` を含まない場合、sync 時に stderr へ `hint: run 'portool init' to install the post-checkout hook`
11. **`ls` 表示**: 列 `PROJECT WORKTREE BRANCH BLOCK STATUS`(2 スペース区切り・列幅揃え)。WORKTREE は `$HOME` を `~` に短縮。branch null は `-`。BLOCK は `3000-3004` 形式。STATUS は pinned → `pinned`、worktree ディレクトリ存在 → `active`、不在 → `stale?`
12. **`ls --json`**: 台帳スキーマと同形の JSON(`{version, range, projects, reservations}`)。デフォルトはカレントプロジェクトのみに projects をフィルタ、`--all` で全件。git リポジトリ外での `ls --all --json` は許容(exit 0)、リポジトリ外で `--all` 無しは exit 1
13. **prune 出力**: 回収 1 件につき stdout 1 行 `pruned <worktree_path> (block <start>-<end>)`。`--dry-run` は `would prune ...`。プロジェクトエントリごと回収は `pruned project <name> (<common_dir>)`
14. **`registry.range`**: 台帳新規作成時に config の pool を記録する情報フィールド。以後の確保は常に現在の config を参照し、`range` フィールドは更新しない
15. **bind チェック**: `std::net::TcpListener::bind(("127.0.0.1", port))`(std のデフォルト、SO_REUSEADDR 付き)。成功 = 空きとみなす。GC の「listen されていない」判定も同じ bind 成功で判定
16. **プロジェクト name 推定**: common_dir の末尾が `.git` → その親ディレクトリ名。それ以外 → common_dir の末尾コンポーネント名
17. **realpath**: `std::fs::canonicalize`。worktree list の各パスは canonicalize 失敗時(消えた worktree)は raw パスで比較
18. **ポート型**: 内部計算は u32 で行い境界チェック後に u16 化(オーバーフロー防止)。pool 上限は 65535 まで許容
19. **flock**: `fs2::FileExt::try_lock_exclusive` を 50ms 間隔でポーリング、10s で exit 4
20. **壊れた台帳**: parse 失敗 → `registry.json.corrupt-<unix秒>` へ rename、stderr に warning、空台帳から再生成(fast path では slow path へフォールスルー)

### 注意すべきエッジケース(実装・テストで必ずカバー)

- `git rev-parse --git-common-dir` は相対パス(`.git`)を返しうる → cwd 基準で絶対化してから canonicalize
- linked worktree から実行しても common_dir は main の `.git` に解決されること
- detached HEAD(FNV キーが worktree パスになる)
- マニフェスト変更: 縮小(現ブロックに収まる → ブロック据え置き)と拡大(収まらない → 解放して再割り当て)の両方
- 同一ブロックが bind 不可(他プロセスが listen)のときスロットをスキップして次へ
- GC: pinned エントリは回収しない / ディレクトリが存在すれば回収しない / ポート listen 中なら回収しない
- 台帳ファイル・ディレクトリ不在(初回実行)で自動作成
- git リポジトリ外での `sync` → exit 1、stderr にエラー
- `.portool.toml` の parse エラー・不正キー・重複オフセット → exit 1

## モジュール構成

```
src/
  main.rs        # clap CLI 定義、コマンドディスパッチ、exit code 変換のみ
  lib.rs         # モジュール公開(統合テスト用に lib + bin 構成)
  error.rs       # Error 型と exit_code() マッピング
  config.rs      # グローバル設定のパースとデフォルト
  manifest.rs    # .portool.toml パース、block_size、env 変数名、manifest_hash
  registry.rs    # 台帳 serde 型 + 純粋な照会/変更ヘルパ
  alloc.rs       # FNV-1a、サブレンジ探索、ブロック割り当て(純関数、bind チェックはクロージャ注入)
  envfile.rs     # .env.portool の内容生成(純関数)
  paths.rs       # XDG パス解決
  store.rs       # 台帳の load(破損処理込み)/ atomic save
  lock.rs        # flock 取得(ポーリング + タイムアウト)
  gitctx.rs      # git コマンドラッパ(common_dir / worktree root / branch / worktree list)
  ports.rs       # 127.0.0.1 bind チェック
  gc.rs          # GC 判定と実行(worktree list / dir 存在 / bind をクロージャ注入で純関数化)
  cmd/
    sync.rs      # fast path / slow path
    init.rs      # フック設置 / .gitignore / sync
    ls.rs        # テーブル / --json / --all
    prune.rs     # prune / --all / --dry-run
tests/
  cli.rs         # E2E: 実バイナリ + 一時 git リポジトリ + XDG_STATE_HOME 隔離
  concurrency.rs # flock を実プロセス並行で叩く統合テスト
```

## Task 1: クレート雛形と純粋コア(config / manifest / alloc / envfile / registry 型)

**目的**: I/O を含まない純粋ロジックをすべて実装しユニットテストで固める。

1. `cargo init` 相当の雛形を手書きで作成:
   - `Cargo.toml`: name/version 0.1.0/edition 2021/description "Passive global port ledger for git worktrees"/license "MIT"/repository "https://github.com/t09tanaka/portool"/readme "README.md"/keywords ["git","worktree","port","ports","dev-tools"]/categories ["command-line-utilities","development-tools"]。依存: clap(derive), serde(derive), serde_json, toml, fs2, sha2, chrono, tempfile。dev-deps: tempfile
   - `src/lib.rs`(モジュール宣言)と `src/main.rs`(clap 定義のみ。サブコマンド `init/sync/ls/prune` と全フラグを Global Constraints どおり定義し、ハンドラは `eprintln!("not implemented"); exit 1` の暫定でよい — Task 3 で置換)
2. `src/error.rs`: `Error` enum(`General(String)` / `SubrangeExhausted` / `PoolExhausted` / `LockTimeout`)、`exit_code()` → 1/2/3/4、`Display`、`From<std::io::Error>` 等
3. `src/config.rs`: `Config { range: (u16,u16), subrange_size: u16, block_align: u16, gc_days: u32 }`。`Default` は Global Constraints の値。`Config::from_toml_str(&str) -> Result<Config>`(部分指定はデフォルトで補完、不正値 — range 逆転・subrange_size 0・block_align 0 — はエラー)
4. `src/manifest.rs`:
   - `Manifest { ports: Vec<(String, u16)> }`(オフセット昇順に正規化)
   - `Manifest::parse(&str) -> Result<Manifest>`: `[ports]` テーブル必須。キー regex `^[a-z][a-z0-9_]*$` 検証、オフセット重複エラー、負値・非整数エラー
   - `block_size(&self, block_align: u16) -> u16` = max(最大オフセット+1, 宣言数) を align 倍数へ切り上げ。マニフェスト無しケース用に `default_block_size(block_align) = block_align`
   - `env_var_name(key) -> String`(大文字化 + `_PORT`)
   - `manifest_hash(bytes: &[u8]) -> String`(SHA-256 hex 先頭 12 文字)
5. `src/registry.rs`: 仕様 §5 のスキーマを serde 型で定義(`Registry`/`ProjectEntry`/`WorktreeEntry`/`Reservation`)。`(u16,u16)` は JSON 配列にシリアライズされること。`Registry::empty(range)` / `all_subranges()` / `all_blocks()`(全プロジェクト + reservations)/ `find_project` / `overlaps(a, b)` 等の純粋ヘルパ
6. `src/alloc.rs`(すべて純関数):
   - `fnv1a_32(bytes: &[u8]) -> u32`(既知テストベクタで検証: 空文字列 → 0x811c9dc5、`"a"` → 0xe40c292c)
   - `find_free_subrange(pool, occupied: &[(u16,u16)], size) -> Option<(u16,u16)>`(先頭から走査、最初の空き)
   - `preferred_slot(branch: Option<&str>, worktree_path: &str, slots: u32) -> u32`(main/master → 0)
   - `allocate_block(subranges: &[(u16,u16)], block_size, branch, worktree_path, occupied: &[(u16,u16)], bind_ok: &mut dyn FnMut((u16,u16)) -> bool) -> Option<(u16,u16)>`(凍結判断 3 の探索順)
7. `src/envfile.rs`: `render(project_name, worktree_path, block, manifest: Option<&Manifest>) -> String`(Global Constraints の厳密形式。manifest None → `PORT=` 1 行)
8. ユニットテスト: 上記全関数。特に block_size 切り上げ(4→5, 5→5, 6→10)、FNV テストベクタ、preferred slot(main→0、feature ブランチのハッシュ分散、detached→パスキー)、subrange 探索(隙間充填・プール枯渇 None)、allocate_block(占有スキップ・bind 失敗スキップ・全滅 None)、envfile バイト厳密一致、registry の serde 往復(仕様 §5 の JSON 例そのままを deserialize できること)
9. `cargo fmt` + `cargo clippy -- -D warnings` + `cargo test` を通してからコミット

**受け入れ基準**: `cargo test` 全緑、clippy 警告ゼロ、仕様 §5 の JSON 例(コメント除去版)が round-trip できる。

## Task 2: I/O 層(paths / store / lock / gitctx / ports / gc)

**目的**: 薄い I/O 層を実装し、実 git リポジトリ・実ファイルでテストする。Task 1 の型・純関数を使う。

1. `src/paths.rs`: `state_dir()` = `$XDG_STATE_HOME`(未設定なら `$HOME/.local/state`)+ `/portool`。`registry_path()` / `lock_path()`(= registry.json.lock)/ `config_path()` = `$XDG_CONFIG_HOME`(未設定なら `$HOME/.config`)+ `/portool/config.toml`。環境変数は毎回読む(テストが XDG_STATE_HOME を差し替えるため)
2. `src/config.rs` に `Config::load() -> Config` を追加: config_path が存在すればパース(パース失敗は stderr warning + デフォルト続行)、無ければデフォルト
3. `src/store.rs`:
   - `load(path) -> LoadResult`: 不在 → `Registry::empty` / parse 失敗 → `registry.json.corrupt-<unix秒>` へ rename + stderr warning + empty(rename 失敗も warning で続行)。fast path 用に「破損していたか」を返す
   - `save(path, &Registry)`: 同一ディレクトリの temp ファイルへ serde_json pretty 書き込み → rename。親ディレクトリは自動作成
4. `src/lock.rs`: `acquire(lock_path, timeout: Duration) -> Result<Lock>`。ロックファイルを create して `fs2::FileExt::try_lock_exclusive` を 50ms ポーリング、タイムアウトで `Error::LockTimeout`。`Drop` で unlock
5. `src/gitctx.rs`: `GitCtx::discover(cwd) -> Result<GitCtx>`:
   - `common_dir`: `git -C <cwd> rev-parse --git-common-dir` → cwd 基準で絶対化 → canonicalize
   - `worktree_root`: `git -C <cwd> rev-parse --show-toplevel` → canonicalize
   - `branch`: `git -C <cwd> symbolic-ref --short -q HEAD` → 失敗/空なら None
   - `project_name`: 凍結判断 16
   - git リポジトリ外 → `Error::General`
   - `worktree_list(&self) -> Result<Vec<PathBuf>>`: `git -C <common_dir_の親 or worktree_root> worktree list --porcelain` の `worktree ` 行をパース、可能なら canonicalize
6. `src/ports.rs`: `port_free(u16) -> bool` / `block_free((u16,u16)) -> bool`(127.0.0.1 bind、全ポート成功で true)
7. `src/gc.rs`: `collect(project: &mut ProjectEntry, live_worktrees: &[PathBuf], dir_exists: &dyn Fn(&Path)->bool, block_unused: &dyn Fn((u16,u16))->bool) -> Vec<(String, (u16,u16))>`(回収した (path, block) を返す純関数寄りの実装。条件は仕様 §8.1 の 3 点 AND)
8. テスト(`tempfile` の一時ディレクトリ使用):
   - store: 不在 → empty / 破損 JSON → corrupt 退避 + empty / save→load round-trip / save が temp+rename であること(同 dir に temp 残骸が無いこと)
   - lock: 二重取得がブロックされ、解放後に取得できる / 短いタイムアウトで LockTimeout
   - gitctx: 一時 git リポジトリ(`git init` + commit)で common_dir/worktree_root/branch を検証。`git worktree add` した linked worktree からも同一 common_dir に解決されること。detached HEAD で branch None
   - ports: listener を立てたポートで false、空きポートで true
   - gc: 3 条件の組み合わせ(pinned / dir 存在 / port 使用中はそれぞれ回収されない)
9. `cargo fmt` + `cargo clippy -- -D warnings` + `cargo test` を通してからコミット

**受け入れ基準**: 全テスト緑。テストが実環境の `~/.local/state` を絶対に触らないこと(全テストで XDG_STATE_HOME 等を一時 dir に向ける)。

## Task 3: コマンド実装(sync / init / ls / prune)と CLI 配線

**目的**: 仕様 §7〜§9 のコマンド 4 種を実装し、main.rs の暫定ハンドラを置き換え、実バイナリの E2E テストを書く。

1. `src/cmd/sync.rs`(仕様 §7 を厳守):
   - fast path: GitCtx 解決 → registry を read-only load(破損検知時は slow path へ)→ 自エントリあり + manifest_hash 一致 + `.env.portool` バイト一致 → 即 exit 0(無出力・無書き込み)
   - slow path: flock(10s)→ 台帳再読込 → 必要なら Config::load の値でサブレンジ確保(凍結判断 3, 4)/ブロック割り当て/manifest 変更時の再割り当て(仕様 §6.4)→ 自プロジェクト GC(§8.1)→ last_seen_at 更新(自エントリ)→ 台帳 atomic save → `.env.portool` atomic write
   - 新規エントリ作成時: branch / manifest_hash / pinned=false / label=null / allocated_at=now / last_seen_at=now
   - フック未設置 hint(凍結判断 10)。出力は凍結判断 9
2. `src/cmd/init.rs`(仕様 §9.1 + 凍結判断 6, 7, 8): フック設置(common_dir/hooks/post-checkout)、.gitignore 追記、sync 実行
3. `src/cmd/ls.rs`(仕様 §9.3 + 凍結判断 11, 12)
4. `src/cmd/prune.rs`(仕様 §8.2, §8.3 + 凍結判断 13): デフォルトはカレントプロジェクトのみ、`--all` は全プロジェクト(common_dir 消滅時はポート未使用確認の上プロジェクトごと回収)。`--dry-run` は台帳を書かない。台帳変更は flock 下で
5. `src/main.rs`: clap ディスパッチを本実装に接続。`Error::exit_code()` で終了。エラーメッセージは `portool: error: ...` 形式で stderr
6. `tests/cli.rs`(E2E、`env!("CARGO_BIN_EXE_portool")` 使用、各テストは一時 HOME/XDG_STATE_HOME/XDG_CONFIG_HOME で完全隔離):
   - git リポジトリ外で sync → exit 1
   - マニフェスト無しリポジトリで sync → `.env.portool` に `PORT=3000`、registry に block [3000,3004]
   - マニフェスト(web=0..db=3)で sync → 4 変数、block [3000,3004]
   - sync 2 回目が no-op(ファイル mtime 不変 & exit 0)
   - `git worktree add` した 2 本目の worktree で sync → 別ブロック、main worktree は block 3000 起点(slot 0)
   - マニフェスト拡大(オフセット 7 追加)で sync → ブロック再割り当て(サイズ 10)
   - マニフェスト縮小で sync → ブロック据え置き、env のみ更新
   - init → post-checkout 生成(実行ビット付き・内容一致)、.gitignore に `.env.portool`、2 回目 init が冪等
   - ls / ls --json の出力形状
   - worktree を削除して sync or prune → エントリ回収。prune --dry-run では消えない
   - detached HEAD で sync → branch null
7. `cargo fmt` + `cargo clippy -- -D warnings` + `cargo test` を通してからコミット

**受け入れ基準**: E2E 全緑。fast path が registry.json / .env.portool に一切書き込まないこと(mtime 検証)。

## Task 4: 並行統合テスト・README・publish 準備・CI
