# portool v0.8.0 — 3巡目外部レビュー対応 設計書

Date: 2026-07-17
Status: Approved（ユーザー承認済み、PR完成まで委任）

## 背景

v0.7.0 公開後の外部レビュー（製品評価 7.2/10）で、「リリース前に必須」として
P1 実装課題4件 + env 優先順位の製品課題1件が指摘された。ユーザー決定により
0.8.0 のスコープは **必須5件 + fault injection テスト** とする。
配布整備（prebuilt/Homebrew）、新コマンド（show/get/why）、JSON 出力拡大、
init --detect、property-based test、TUI 系は今回のスコープ外。

### ユーザー決定事項

1. スコープ: 必須5件 + fault injection
2. exec のデフォルト bind チェック警告は廃止し、`--check-ports` で opt-in
3. in-place 拡張不可 & 現ブロックでリスナー稼働中の sync は移動を**拒否**し、
   明示操作（サーバー停止して再 sync / `portool reallocate`）を案内する

## 1. manifest 拡張時の in-place 優先（P1-1）

### 問題

現状の `sync` slow path は、manifest 拡張で現ブロックに収まらなくなると
`allocate_or_reuse_block`（existing_block を occupied から除外した通常割当）を
呼ぶだけ。preferred slot は `hash % num_slots` で `num_slots` がブロックサイズに
依存するため、サイズが変わると同じ branch でも別位置に飛びやすい。さらに
`bind_ok` がブロック全体を bind チェックするため、自分のサーバーが旧ブロックで
稼働中だと「旧ブロックを包含する拡張候補」は必ず失敗し、稼働中サーバーを
置き去りにした移動（worktree 内の新旧ポート分裂）が起きる。

### 設計

`sync` slow path の「manifest 拡張でサイズ超過」分岐で、通常アロケータの前に
**in-place 拡張候補を最優先で試す**:

1. 候補 = `(現start, 現start + 新size - 1)`（開始ポート維持）
2. プール `(range.0, range.1)` に収まること
3. 台帳の他エントリ・予約と重複しないこと（自エントリの現ブロック・pending は除外）
4. bind チェックは**追加された末尾範囲 `(旧end+1, 新end)` のみ**
   （旧ブロック部分は自分のプロセスが使っていて当然なので検査しない）
5. 成功時は grow として扱い generation 据え置き。二相更新（pending 予約 →
   env 書き込み → finalize）は従来どおり適用する
   （v0.6 の pending validate は自エントリ block との重複 = grow-in-place を既に許容）

拡張不可（プール外 / 重複 / 末尾 bind 失敗）の場合のみ移動へフォールバック。
ただし移動直前に `ports::block_free(旧block)` を確認し、**false（現ブロックで
何か稼働中）なら移動せずエラー**:

- エラーメッセージ: 現ブロックで稼働中のプロセスがあり自動移動は危険である旨と、
  「サーバーを停止して再度 `portool sync`、または `portool reallocate` を明示実行」
  の案内
- hook 経由の呼び出しでも git 操作は失敗しない（hook スクリプトの exit 0 保証は既存）
- `portool reallocate` は明示コマンドなので従来どおり無条件で移動可能

### エッジケース

- 現ブロックがプール末尾際で拡張がプール外に出る → 移動フォールバック
- 拡張候補が自分の pending_block と重なる → resolve_pending_move が先に走るため
  slow path 到達時点で pending は解消済み
- block_align との関係: 現 start は割当時に align 済みであり、in-place 拡張は
  start を動かさないので align 制約を破らない
- 縮小（新size ≤ 現幅）は従来どおり keep-in-place（既存挙動、変更なし）
- 拡張成功後の env 再レンダリングは新ブロック幅で全サービスを含む

## 2. exec のデフォルト bind チェック廃止（P1-2）

### 設計

- **デフォルト**: bind チェックを行わず静かに起動。
  `portool exec -- npm run frontend` と `portool exec -- npm run backend` の
  並行起動で警告ノイズが出なくなる
- `--check-ports`: 従来の advisory 警告（ブロック全体 bind チェック → stderr 警告）
- `--strict`: `--check-ports` を含意し、競合時にエラー終了（既存の意味を維持）
- `--reallocate-on-conflict`: チェックを含意し競合時に別ブロックへ移動。
  ヘルプ文と実行時警告に「現ブロックで稼働中のプロセスがある場合、worktree
  全体が別ポートへ移り新旧分裂の危険がある」旨を明記する

### 互換性

CLI デフォルト挙動の変更 = 互換性境界の変更。CHANGELOG に breaking として記載。
`--strict` / `--reallocate-on-conflict` 単独指定は従来どおり動く（チェック含意）。

## 3. 永続化の fsync 強化（P1-3）

### 設計

atomic 書き込みを一本化し、durable にする:

- `write_atomic`（新 util、`store.rs` に集約）:
  temp file 書き込み → `sync_all` → rename（persist）→ 親ディレクトリを
  open して `sync_all`
- 適用経路: ① `.env.portool` 書き込み（sync.rs の現 `write_atomic` を置換）
  ② 台帳 `registry.json` 保存 ③ バックアップ `.bak`
- バックアップは `fs::copy` 直書きをやめ、`registry.json.bak.tmp` へ書く →
  fsync → rename → 親 dir fsync（コピー途中クラッシュで旧バックアップが
  部分破壊される問題を解消）
- バックアップ失敗が warning（コマンド失敗にしない）なのは現状維持
- macOS では `File::sync_all` が fcntl F_FULLFSYNC まで行わない点は許容
  （プロセスクラッシュ + 通常の電源断レベルを狙う。ドキュメントで誇張しない）

### エッジケース

- 親 dir fsync が失敗するプラットフォーム（一部 FS）→ エラーにせず warning
  で続行（rename 自体は成功している）
- tempfile::persist の既存エラーハンドリングは維持

## 4. 人間向け出力のサニタイズと Unicode 幅（P1-4）

### 設計

新モジュール `src/display.rs`:

- `sanitize(&str) -> String`: 制御文字の無害化
  - ESC (U+001B) およびその他の C0/C1 制御文字 → U+FFFD に置換
  - 改行 (\n) → リテラル `\n`（バックスラッシュ + n の2文字）、\r・\t も同様の可視化
- `width(&str) -> usize`: `unicode-width` crate による表示幅
- 適用先:
  - `ls` テーブル全セル（project / worktree / branch / block / status、および
    label を含むあらゆる台帳由来文字列）
  - 予約ラベル行（`reserved X-Y label`）
  - `sync` / `reallocate` の println に含まれる worktree パス
  - JSON 出力は無加工（serde のエスケープに任せる、現状維持）
- `ls` のカラム幅計算を `String::len()` から `display::width()` ベースへ変更
- 長パスの中央省略は今回やらない（YAGNI）

依存追加: `unicode-width` のみ。

## 5. env ファイル優先順位オプション

### 設計

`portool exec --env-file-overrides` を新設:

- 指定時の優先順位: 親環境 < 前の env ファイル < 後の env ファイル < portool 変数
- 未指定時は現行どおり: 前の env ファイル < 後の env ファイル < 親環境 < portool 変数
- 実装: `compose()` にフラグを渡し、指定時は parent 挿入を file より先に行う
  （file エントリが後勝ちで parent を上書き）。file 由来の勝者は to_set に
  含まれ child に明示 set される。env ファイルに現れない親変数は従来どおり
  そのまま継承（非 UTF-8 値のバイト保全ロジックも維持）
- `--ignore-parent-env`（env_clear 系）は作らない — レビューの実害シナリオ
  （`-e .env.test` 指定が親の `DATABASE_URL` に負ける）はこのフラグで解決する

### エッジケース

- `${NAME}` 展開: file 値が parent を上書きした場合、他の file 行からの参照は
  file 値で解決される（resolve は合成後の map を見るため自然に成立）
- portool 変数は常に最上位（フラグの影響を受けない）

## 6. fault injection テスト

### 設計

- `src/fault.rs`: `pub fn point(name: &str)` — 環境変数
  `PORTOOL_TEST_FAULT` が `name` と一致したら `std::process::abort()`。
  本体は `#[cfg(debug_assertions)]` ガードで、リリースビルドでは no-op
  （テストの実バイナリは debug プロファイルなので有効）
- 設置地点（すべて sync/reallocate/store 内）:
  1. `after_pending_save` — pending 台帳保存後
  2. `before_env_write` — env 書き込み直前
  3. `after_env_write` — env 書き込み直後（finalize 前）
  4. `after_final_save` — 最終台帳保存後・バックアップ前
  5. `during_backup` — バックアップ temp 書き込み後 rename 前
- E2E テスト（`tests/`）: 各地点で `PORTOOL_TEST_FAULT` を設定して実バイナリを
  abort させた後、`sync` / `doctor` を実行し以下の不変条件を検証:
  - 台帳のブロック重複なし（validate が通る）
  - 再 sync 後に env と台帳が一致（roll-forward / roll-back の完遂）
  - バックアップが `load_strict` で読める（during_backup 後も旧 .bak が無傷）
- ポートレンジ: 既存規約どおり専用レンジを確保（使用済み:
  3000s/3800s/3900s/4200-4319, 17300-17699, 18000-18299, 18500-18599,
  18600-18609, 18700-18799, 18800-18809 — 新規テストは空きを grep で確認して選定）
