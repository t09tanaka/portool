# portool Worktree Identity Metadata 仕様書
## 1. 目的
各Git worktreeに対して、ポート割り当てとは独立した安定識別子を生成し、`.env.portool` へ出力する。
主な用途は以下。
- Docker Composeのproject名
- container / network / volumeの名前空間
- worktree固有のキャッシュ・ログ・一時ディレクトリ
- その他、ポート以外のローカルリソース分離
portool自身はDockerやプロセスを管理しない。汎用的な識別子を提供し、利用方法は消費側へ委ねる。
## 2. 背景
現在のportoolはworktreeごとに異なるポートブロックを割り当てるが、Docker Composeではホストポートだけを変えても、project名・container・network・volumeが共有される可能性がある。
以下を識別子として利用してはならない。
- ポート番号
  - manifest変更、registry再作成、再割り当てで変わる可能性がある
- ブランチ名
  - rename、detached HEAD、同名ブランチ、使用不能文字の問題がある
- 表示用project名
  - 同名リポジトリや複数cloneを区別できない
## 3. スコープ
`.env.portool` に以下を常に出力する。
```dotenv
PORTOOL_PROJECT_ID=<project-id>
PORTOOL_WORKTREE_ID=<worktree-id>
```
- `PORTOOL_PROJECT_ID`
  - 同一Git common-dirを共有する全worktreeで同じ
  - 別cloneは別ID
- `PORTOOL_WORKTREE_ID`
  - worktreeごとに異なる
  - 同一マシン上でグローバルな名前空間として利用可能
`PORTOOL_*` はportool予約環境変数プレフィックスとする。
## 4. Non-goals
以下は実装しない。
- `COMPOSE_PROJECT_NAME` の直接生成
- Docker Composeの起動・停止
- container / network / volumeの管理
- 任意のenvテンプレート機能
- 人間向けslugの生成
- ブランチ名を使った識別子
- IDのregistry保存
- repository移動後も同じIDを維持する仕組み
- 異なるマシン・異なるclone間で同じIDを生成する仕組み
既存仕様の「DB / Compose projectの隔離はportoolの責務外」は維持する。今回追加するのは、隔離に利用可能な汎用メタデータだけとする。
## 5. 同一性の定義
既存のGit同一性をそのまま利用する。
```text
common_dir    = canonicalize(git rev-parse --git-common-dir)
worktree_root = canonicalize(git rev-parse --show-toplevel)
```
### Project ID
```text
SHA-256(
  "portool-project-id-v1\0"
  + raw_bytes(common_dir)
)
```
SHA-256結果の先頭16文字を、小文字hexで出力する。
### Worktree ID
```text
SHA-256(
  "portool-worktree-id-v1\0"
  + raw_bytes(common_dir)
  + "\0"
  + raw_bytes(worktree_root)
)
```
SHA-256結果の先頭16文字を、小文字hexで出力する。
### 固定事項
- 文字数は16文字
- 文字種は `[0-9a-f]{16}`
- salt、乱数、時刻、branch、port、registry内容を含めない
- macOS/Linuxではcanonical pathのOS raw bytesを使用する
- prefix中の `v1` はハッシュ入力のdomain/version separator
- NULはpathに含まれないため、区切り文字として安全に利用できる
## 6. 既知テストベクタ
入力:
```text
common_dir    = /home/user/dev/myapp/.git
worktree_root = /home/user/dev/myapp
```
期待値:
```dotenv
PORTOOL_PROJECT_ID=abe4c983b7295f37
PORTOOL_WORKTREE_ID=740da24128c5e2f7
```
linked worktree:
```text
common_dir    = /home/user/dev/myapp/.git
worktree_root = /home/user/dev/myapp-wt/feat-api
```
期待値:
```dotenv
PORTOOL_PROJECT_ID=abe4c983b7295f37
PORTOOL_WORKTREE_ID=39c020aea86d3272
```
## 7. `.env.portool` 出力形式
識別子はヘッダーコメントの直後、ポート変数の前に出力する。
### manifestあり
```dotenv
# generated by portool — DO NOT EDIT
# block: 3000-3004  project: myapp  worktree: /home/user/dev/myapp
PORTOOL_PROJECT_ID=abe4c983b7295f37
PORTOOL_WORKTREE_ID=740da24128c5e2f7
WEB_PORT=3000
API_PORT=3001
DB_PORT=3002
```
### manifestなし
```dotenv
# generated by portool — DO NOT EDIT
# block: 3000-3004  project: myapp  worktree: /home/user/dev/myapp
PORTOOL_PROJECT_ID=abe4c983b7295f37
PORTOOL_WORKTREE_ID=740da24128c5e2f7
PORT=3000
```
識別子はmanifestの有無にかかわらず必ず出力する。
値は引用符で囲まない。ファイル末尾には従来どおり改行を付ける。
## 8. 安定性
以下の操作ではIDを変更しない。
- ブランチのcheckout
- ブランチrename
- detached HEADへの移行
- `.portool.toml` の変更
- ポートブロックの再割り当て
- portool registryの削除・再生成
- portoolのglobal config変更
- 同じパスへのworktree再作成
以下の場合はIDが変わる。
- repositoryまたはworktreeの移動
- 別ディレクトリへのclone
- worktreeを別パスで再作成
- Git common-dirが変わる操作
これは既存portoolの「common-dirとworktree canonical pathを同一性とする」仕様に一致する。
## 9. アルゴリズムの互換性
IDはvolumeやキャッシュの名前に使われる可能性があるため、一度リリースしたアルゴリズムを変更してはならない。
将来アルゴリズム変更が必要になった場合、既存変数の値を黙って変更せず、次のいずれかを行う。
- 新しいversion付き変数を追加する
- 明示的なmigration機構を提供する
- メジャーな互換性変更として告知する
ハッシュ長、domain separator、入力順、canonicalization方式も互換性契約に含む。
## 10. Registryへの影響
registryスキーマは変更しない。
IDは以下から純粋に導出できる。
- `GitCtx.common_dir`
- `GitCtx.worktree_root`
registryへ保存しない理由:
- registry削除後も同じIDを再生成できる
- staleデータとの同期問題を避けられる
- schema migrationが不要
- IDとGit同一性の二重管理を避けられる
`portool ls --json` の出力も今回変更しない。
## 11. Sync挙動
新バージョンで初めて `portool sync` を実行すると、既存 `.env.portool` にはID行がないためfast pathのバイト一致判定が失敗する。
その後のslow pathでは:
- 既存ポートブロックを維持する
- manifest hashを変更しない
- registry schemaを変更しない
- ID行を追加した `.env.portool` をatomic writeする
更新後、2回目以降のsyncは従来どおりlock-free fast pathで終了する。
## 12. 後方互換性
以下は変更しない。
- CLIコマンド
- exit code
- `.portool.toml` 形式
- ポート割り当て
- registry形式
- port変数名
- hook動作
追加されるのは `.env.portool` の2変数のみ。
dotenvとしてsourceする利用者には後方互換。ただし「コメント以外の全行がポート変数である」と仮定した独自parserには影響があるため、READMEに次を明記する。
> Consumers must ignore unknown `PORTOOL_*` metadata variables and must not assume every variable ends with `_PORT`.
この変更はpatchではなく `0.2.0` としてリリースする。
異なるバージョンのportoolを交互に実行すると、旧版がID行を削除し、新版が再追加する可能性がある。通常は単一バイナリを利用する前提とし、この挙動は許容する。
## 13. 利用例
### Docker Compose
```yaml
name: esim-commerce-${PORTOOL_WORKTREE_ID:-default}
```
起動:
```sh
docker compose --env-file .env.portool up
```
### 開発DBとテストDBを別projectにする場合
開発用:
```yaml
name: esim-commerce-dev-${PORTOOL_WORKTREE_ID:-default}
```
テスト用:
```yaml
name: esim-commerce-test-${PORTOOL_WORKTREE_ID:-default}
```
portool未導入時は `default` へフォールバックできる。
## 14. エッジケース
| ケース | 挙動 |
|---|---|
| main worktreeとlinked worktree | Project IDは同じ、Worktree IDは異なる |
| detached HEAD | IDは変わらない |
| branch rename | IDは変わらない |
| 同じbranchを別pathで再作成 | Worktree IDは異なる |
| 同じpathへworktreeを再作成 | Worktree IDは同じ |
| 同一repositoryの別clone | Project ID、Worktree IDとも異なる |
| repositoryを移動 | 両IDが変わる |
| worktreeだけ移動 | Worktree IDのみ変わる |
| manifest変更によるport再割り当て | 両IDとも変わらない |
| registry削除 | 両IDとも変わらない |
| 非UTF-8 path | Unixのraw path bytesをhashする |
| hash衝突 | 64bit truncationの確率上は許容し、registry照合は行わない |
IDは秘密情報ではない。元pathのハッシュではあるが、推測可能なpathに対する辞書攻撃は可能であるため、認証トークンなどには使用しない。
なお、現在の `.env.portool` ヘッダー自体がworktree pathを平文で含んでいるため、この追加によって新しい機密情報区分が発生するわけではない。
## 15. テスト要件
### Unit tests
- Project ID既知ベクタ
- Worktree ID既知ベクタ
- 同一common-dirで異なるworktree pathは異なるID
- 異なるcommon-dirでは異なるProject ID
- branch情報がID計算へ影響しない
- 出力が常に16文字の小文字hex
- manifestあり・なし両方のenvfile exact-byte test
- ID行がport変数より先に出力される
- trailing newlineが維持される
### CLI/E2E tests
- mainとlinked worktreeでProject IDが一致する
- mainとlinked worktreeでWorktree IDが異なる
- checkout後もWorktree IDが変わらない
- detached HEADでもWorktree IDが変わらない
- manifest拡大によるblock再割り当て後もIDが変わらない
- registry削除・再sync後もIDが変わらない
- 旧形式の `.env.portool` が最初のsyncで更新される
- 更新後の2回目syncが完全なno-opで、env/registryのmtimeを変更しない
## 16. 受け入れ基準
- `.env.portool` に2つの識別子が常に出力される
- 同一プロジェクト内の全worktreeでProject IDが一致する
- 各worktreeのWorktree IDが異なる
- IDがbranch・port・manifest・registryから独立している
- portoolがDocker固有機能を持たない
- registry schemaとCLIに変更がない
- v0.1の既存テストを更新後もすべて通過する
- 新規unit/E2Eテストがすべて通過する
- READMEに識別子、安定性、Compose利用例、未知変数を無視する要件が記載される
